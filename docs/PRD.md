# 专利文件智能审查系统 PRD

## 1. 文档信息

- 产品名称：专利文件智能审查系统
- 项目代号：patent_check_agent
- 文档类型：产品需求文档（PRD）
- 文档版本：v1.0
- 编写日期：2026-06-23
- 目标部署环境：阿里云 ECS + 公司域名 + DNS 解析

## 2. 项目背景

实验室需要一个内部网站，用于辅助检查专利申请文件中的权利要求书、说明书、附图说明和摘要。用户登录后上传待审查文件，系统先从文件中抽取文本，再调用公司提供的 GPT-5.5 API，并加载专利检查规则模板，生成结构化审查报告。

现有检查规则来自 `check-patent.md`，其核心流程为两阶段耦合审查：

1. 第一阶段：检查权利要求书，拆解技术特征，并生成“需说明书解释项清单”。
2. 第二阶段：基于第一阶段清单检查说明书、附图说明和摘要，判断说明书是否对权利要求技术特征给出充分解释。

## 3. 产品目标

### 3.1 核心目标

- 提供一个实验室内部可访问的网站。
- 支持固定分账号登录，不开放注册。
- 支持用户上传专利文件并自动抽取文本内容。
- 后端调用 GPT-5.5 API 执行专利检查规则。
- 返回结构化、可阅读、可复制、可下载的审查报告。
- 支持审查历史记录，便于用户查看过往任务。

### 3.2 非目标

第一版不实现以下能力：

- 不提供公开注册。
- 不提供多人协同编辑。
- 不直接在线编辑 Word/PDF。
- 不提供正式法律意见或自动修改专利文件。
- 不实现复杂后台运营系统。
- 不把 Codex CLI 或 Codex Agent 作为生产运行时依赖。

## 4. 用户与角色

### 4.1 普通用户

普通用户为实验室成员或公司内部授权人员。

权限：

- 登录系统。
- 上传专利文件并提交审查任务。
- 查看本人提交的审查任务。
- 查看、复制、下载本人任务的审查报告。

### 4.2 管理员

管理员为系统维护人员。

权限：

- 拥有普通用户所有权限。
- 创建、停用、重置内部账号。
- 查看全部审查任务。
- 查看系统调用状态和失败原因。

第一版不提供管理员页面，账号通过数据库种子脚本预置和维护。首批预置 3 个内部账号。

## 5. 用户流程

### 5.1 登录流程

1. 用户访问系统域名。
2. 系统展示登录页，包含实验室 Logo、账号、密码输入框。
3. 用户输入账号密码并提交。
4. 后端校验账号状态和密码哈希。
5. 登录成功后进入审查页面。
6. 登录失败时提示账号或密码错误。

### 5.2 专利审查流程

1. 用户进入审查页面。
2. 用户上传以下文件：
  - 权利要求书文件
  - 说明书文件
  - 附图说明文件
  - 摘要文件
3. 用户可选填写技术领域，例如人工智能、通信、机械、材料、化学等。
4. 用户点击“开始审查”。
5. 后端接收文件，保存文件元数据，并抽取文件文本。
6. 后端创建审查任务，任务状态为 `pending`。
7. Worker 异步执行审查任务，任务状态变为 `running`。
8. Worker 调用 GPT-5.5 API，生成阶段性结果和最终报告。
9. 任务成功后状态变为 `succeeded`。
10. 前端展示审查报告，支持复制和下载。
11. 如果任务失败，前端展示失败原因和重试入口。

### 5.3 历史记录流程

1. 用户进入历史记录页面。
2. 系统展示用户提交过的任务列表。
3. 用户可按状态、时间、关键词筛选。
4. 用户点击某条任务进入详情页。
5. 系统展示该任务的上传文件信息、抽取文本长度、状态和审查报告。

## 6. 功能需求

### 6.1 登录与鉴权

#### 6.1.1 登录页面

页面元素：

- 实验室 Logo。
- 系统名称。
- 账号输入框。
- 密码输入框。
- 登录按钮。
- 错误提示区域。

验收标准：

- 未登录用户访问业务页面时自动跳转登录页。
- 登录成功后跳转审查页面。
- 登录失败时不泄露具体原因，只提示账号或密码错误。

#### 6.1.2 账号机制

需求：

- 系统不开放注册。
- 账号由管理员预置。
- 密码必须使用哈希存储。
- 支持账号启用和停用。
- 支持普通用户和管理员两类角色。

验收标准：

- 数据库中不保存明文密码。
- 停用账号无法登录。
- 普通用户不能查看他人的审查任务。

### 6.2 专利文件上传

#### 6.2.1 上传字段

必填字段：

- 权利要求书文件。
- 说明书文件。
- 附图说明文件。

可选字段：

- 摘要文件。
- 技术领域。
- 任务标题。

第一版支持文件格式：

- `.pdf`

验收标准：

- 权利要求书文件和说明书文件为空时不能提交。
- 文件格式不支持时给出明确提示。
- 文件大小超出系统限制时给出明确提示。
- 文件文本抽取失败时给出明确提示。
- 成功提交后立即返回任务 ID，不阻塞等待模型完成。

#### 6.2.2 文件与文本限制

第一版文件与文本技术限制：

- 单个文件大小不超过 20 MB。
- 单个任务最多上传 4 个文件。
- 单个任务总输入文本不超过 200,000 个中文字符。

限制值应作为后端配置项，便于上线后调整。第一版不对账号任务总数、账号并发任务数、每日调用量设置硬限制。

#### 6.2.3 文件文本抽取

需求：

- 后端在创建任务前从上传文件中抽取文本。
- `.pdf` 第一版只要求支持可复制文本型 PDF，不支持扫描件 OCR。
- 抽取后的文本进入审查流程。
- 第一版不长期保存用户上传的原始专利文件和完整抽取文本；任务完成或失败后，按清理策略删除原始文件和过程文本，仅保留必要元数据、任务状态、错误信息、阶段结果和最终报告。

验收标准：

- 上传可复制文本型 `.pdf` 后可以正确抽取文本。
- 扫描件 PDF 抽取失败时提示用户上传可复制文本型 PDF。
- 历史任务详情页可以展示文件元数据、文本长度、状态和最终报告，不展示完整抽取文本。

### 6.3 审查任务

#### 6.3.1 任务状态

任务状态包括：

- `pending`：已提交，等待执行。
- `running`：正在审查。
- `succeeded`：审查成功。
- `failed`：审查失败。
- `cancelled`：已取消，第一版可不实现取消按钮。

验收标准：

- 前端可以查询任务状态。
- 后端记录任务创建时间、开始时间、结束时间。
- 失败任务记录失败原因。

#### 6.3.2 异步执行

需求：

- 审查任务必须异步执行。
- 后端 API 负责创建任务。
- Worker 负责模型调用和报告生成。
- 前端通过轮询或 SSE 获取任务状态。

第一版推荐使用轮询，后续可升级为 SSE。

验收标准：

- 用户提交任务后页面不超时。
- 模型调用较慢时用户可以看到进行中状态。
- 任务失败不会导致后端服务崩溃。

### 6.4 GPT-5.5 审查能力

#### 6.4.1 审查模板管理

需求：

- 将 `check-patent.md` 放入后端项目中的 `prompts/check_patent.md`。
- 后端读取该模板作为审查规则。
- 模板变更应通过 Git 管理。

验收标准：

- 模型调用使用固定版本的审查模板。
- 审查报告包含模板要求的核心结构。

#### 6.4.2 两阶段审查流程

第一阶段输入：

- 权利要求书。
- 技术领域。
- 专利检查模板中第一阶段规则。

第一阶段输出：

- 权利要求书通过项。
- 权利要求书问题项。
- 技术特征分解与需说明书解释项清单。

第二阶段输入：

- 说明书。
- 附图说明。
- 摘要。
- 第一阶段生成的需说明书解释项清单。
- 专利检查模板中第二阶段规则。

第二阶段输出：

- 说明书通过项。
- 说明书问题项。
- 权要特征-说明书解释对应检查表。
- 附图与摘要检查结果。
- 总体评价。

验收标准：

- 最终报告必须包含“技术特征分解与需说明书解释项清单”。
- 最终报告必须包含“权要特征-说明书解释对应检查表”。
- 问题项需要包含严重程度、涉及条款、建议修改和原文位置。

#### 6.4.3 模型调用策略

推荐策略：

- 后端直接调用 GPT-5.5 API。
- 不在生产主链路中直接调用 Codex CLI。
- API key 仅保存在服务器环境变量中。
- 设置模型调用超时、重试和错误记录。

生产环境不调用 Codex CLI 或 Codex Agent。

### 6.5 报告展示

#### 6.5.1 报告内容

报告应包含：

- 第一阶段：权利要求书检查与特征分解。
- 第二阶段：说明书检查。
- 说明书附图与摘要检查。
- 总体评价。
- 主要风险点。
- 优先修改建议 Top 3。

验收标准：

- 报告支持 Markdown 渲染。
- 表格正常展示。
- 长报告页面可滚动。
- 用户可以复制完整报告。
- 用户可以下载 Markdown 文件。

#### 6.5.2 错误展示

常见错误：

- 抽取文本为空。
- 抽取文本过长。
- 文件格式不支持。
- 文件文本抽取失败。
- 模型调用超时。
- 模型服务不可用。
- API key 配置错误。
- 任务执行异常。

验收标准：

- 前端展示用户可理解的错误文案。
- 后端日志记录详细错误信息。
- 不向前端暴露 API key、堆栈详情或服务器敏感路径。

### 6.6 历史记录

需求：

- 用户可查看本人提交的任务。
- 列表展示任务标题、状态、创建时间、完成时间。
- 详情页展示上传文件元数据、抽取文本长度、状态、失败原因和最终报告。

验收标准：

- 普通用户不能访问他人的任务详情。

## 8. 技术方案

### 8.1 推荐技术栈

前端：

- Next.js
- React
- TypeScript
- Tailwind CSS
- Markdown 渲染组件

后端：

- FastAPI
- Python
- SQLAlchemy
- Alembic
- Pydantic

任务队列：

- Redis
- RQ 或 Celery

数据库：

- PostgreSQL

部署：

- Docker
- Docker Compose
- Nginx
- HTTPS
- 阿里云 ECS

### 8.2 服务划分

推荐服务：

- `frontend`：前端页面。
- `backend`：API 服务。
- `worker`：异步审查任务执行器。
- `postgres`：业务数据库。
- `redis`：任务队列和缓存。
- `nginx`：反向代理和 HTTPS 入口。

### 8.3 数据流

```text
用户登录
  -> 前端提交账号密码
  -> 后端校验用户
  -> 写入登录态

用户提交审查
  -> 前端调用创建任务接口
  -> 后端接收上传文件
  -> 后端保存文件元数据并抽取文本
  -> 后端临时保存抽取文本并记录任务状态
  -> 后端投递 Redis 队列
  -> Worker 拉取任务
  -> Worker 调用 GPT-5.5 API
  -> Worker 保存阶段结果和最终报告
  -> Worker 清理原始文件和过程文本
  -> 前端轮询任务状态
  -> 前端展示报告
```

## 10. API 需求

### 10.1 认证接口

```text
POST /api/auth/login
POST /api/auth/logout
GET  /api/auth/me
```

### 10.2 审查任务接口

```text
POST /api/patent-checks
GET  /api/patent-checks
GET  /api/patent-checks/{task_id}
GET  /api/patent-checks/{task_id}/report
POST /api/patent-checks/{task_id}/retry
```

`POST /api/patent-checks` 使用 `multipart/form-data`，包含任务标题、技术领域和上传文件。

第一版可暂不实现取消任务。

### 10.3 管理接口

```text
GET  /api/admin/users
POST /api/admin/users
PATCH /api/admin/users/{user_id}
POST /api/admin/users/{user_id}/reset-password
GET  /api/admin/patent-checks
```

第一版不实现管理员页面，账号管理通过种子脚本完成。管理接口可作为后续阶段能力。

## 11. 安全需求

- API key 不得出现在前端代码、浏览器网络响应、GitHub 仓库中。
- `.env` 文件不得提交 Git。
- 密码必须哈希存储。
- 登录态使用 HttpOnly Cookie 或安全 JWT 方案。
- 所有业务接口必须鉴权。
- 普通用户只能访问自己的任务。
- 后端错误响应不得暴露堆栈、服务器路径、环境变量。
- 生产环境必须启用 HTTPS。
- 第一版不设置账号任务总数、并发任务数和每日调用量硬限制。
- 建议记录关键操作日志。

## 12. 非功能需求

### 12.1 性能

- 登录接口响应时间应小于 1 秒。
- 创建审查任务接口响应时间应小于 2 秒。
- 审查任务异步完成，不要求同步返回。
- 大文本任务应显示运行中状态。

### 12.2 可用性

- 模型调用失败时，任务状态应变为 `failed`。
- 用户可以查看失败原因并重试。
- Worker 崩溃后重启不应丢失已保存任务。

### 12.3 可维护性

- 审查模板文件版本化管理。
- 前后端分层清晰。
- 模型调用封装为独立模块。
- 每个代码文件头部需要注释说明该文件用途。
- 环境依赖问题应优先解决，不通过绕过方式处理。

### 12.4 可部署性

- 本地开发环境使用 Docker Compose。
- 生产环境使用 Docker Compose 部署到阿里云 ECS。
- 支持通过 `.env` 切换本地和生产配置。

## 13. 部署方案

### 13.1 开发流程

```text
本地开发
  -> Git 管理
  -> GitHub 私有仓库
  -> 阿里云服务器部署
```

### 13.2 本地开发

- 本地启动前端、后端、PostgreSQL、Redis。
- 本地 `.env` 配置测试 API key 和数据库连接。
- 本地跑通登录、提交任务、生成报告流程。

### 13.3 GitHub 管理

- 使用 GitHub private repository。
- `main` 分支保存稳定版本。
- 功能开发使用 feature 分支。
- `.env`、密钥、生产配置不得提交。
- 提供 `.env.example`。

### 13.4 阿里云部署

步骤：

1. ECS 安装 Docker 和 Docker Compose。
2. 配置服务器安全组，只开放必要端口。
3. 拉取 GitHub 私有仓库代码。
4. 在服务器写入生产 `.env`。
5. 启动 Docker Compose 服务。
6. 配置 Nginx 域名反向代理。
7. 配置 HTTPS 证书。
8. 将阿里云 DNS 解析到 ECS 公网 IP。
9. 验证登录、提交任务、报告生成和历史记录。

## 14. 验收标准

### 14.1 MVP 验收

- 用户可以通过账号密码登录。
- 未登录用户无法访问审查页面。
- 用户可以上传权利要求书文件和说明书文件。
- 系统可以从上传文件中抽取用于审查的文本。
- 系统可以异步创建审查任务。
- Worker 可以调用 GPT-5.5 API 生成报告。
- 报告包含两阶段审查内容。
- 报告包含技术特征分解清单。
- 报告包含权要特征与说明书解释对应检查表。
- 用户可以查看历史任务。
- 普通用户不能查看他人任务。
- 系统可以部署到阿里云域名下通过 HTTPS 访问。

### 14.2 上线前检查

- `.env` 未提交 Git。
- API key 未暴露到前端。
- 数据库中无明文密码。
- Docker Compose 可以一键启动服务。
- Nginx HTTPS 正常。
- 模型调用失败有错误处理。
- 上传文件有格式和大小限制。
- 抽取后的长文本有长度限制。
- 日志中不记录完整 API key。

## 16. 风险与应对


| 风险                   | 影响         | 应对                                |
| -------------------- | ---------- | --------------------------------- |
| 专利文件过大或抽取文本过长        | 报告不完整或调用失败 | 设置文件大小和文本长度限制，必要时分段审查             |
| 上传文件无法抽取文本           | 用户无法提交审查   | 明确支持可复制文本型 PDF，扫描件 PDF 提示暂不支持 OCR |
| 模型输出不稳定              | 报告格式难解析    | 使用固定模板和结构化输出约束                    |
| API 调用耗时较长           | 前端请求超时     | 使用异步任务队列                          |
| API key 泄露           | 安全风险       | 仅保存在后端环境变量                        |
| 服务器网络访问模型 API 不稳定    | 任务失败       | 加重试、超时、错误提示                       |
| 用户误以为报告是法律结论         | 合规风险       | 页面提示报告仅作辅助审查参考                    |


## 17. 已确认决策

1. 允许后端直接调用 GPT-5.5 API，并将 `check-patent.md` 作为审查模板使用。
2. 生产环境不必须调用 Codex CLI 或 Codex Agent。
3. 第一批预置 3 个内部账号。
4. 第一版不需要管理员页面，账号管理使用数据库种子脚本。
5. 实验室 Logo 使用 `icon/实验室图标.jpg`，前端按页面空间等比展示。
6. 域名由阿里云提供，按可直接绑定阿里云 ECS 设计；上线前确认备案状态、DNS 解析和 HTTPS 证书。
7. 第一版审查报告只提供 Markdown 展示、复制和下载，不提供 PDF 导出。
8. 第一版只支持 PDF 上传，并优先支持可复制文本型 PDF，不支持扫描件 PDF OCR。
9. 第一版不长期保存用户上传的原始专利文件和完整抽取文本。
10. 第一版不对任务数量、并发数、每日调用量做账号级硬限制。
